The success or failure (https://web.archive.org/web/20120611103012/http://www.cshblogs.org/cshprotocols/2008/02/14/why-web-20-is-failing-in-biology/?) of Web 2.0 efforts for scientists depends to a large extend on the availability of cool applications that make the everyday life of a scientist easier. Many of these applications of course already exist, but I would argue that there is a lot of room for improvement. And I would also argue that in a lot of cases we just have to take the example of the Web 2.0 world and adapt it to the needs of scientists â€“ Nature Network itself would be an example of this approach.

Meetings and Seminars

Scientific meetings and seminars are one example were we can do better. Web 2.0 is an ideal approach for this, and Upcoming (https://web.archive.org/web/20120611103012/http://upcoming.yahoo.com/?) is the classic application. There are of course a number of websites that list meetings and seminars for scientists, but they either focus on the big meetings or list just the seminars of a particular institution. Look at the discussion How to find a science event in Berlin (https://web.archive.org/web/20120611103012/http://network.nature.com/forums/berlin/1360?) in the Nature Network Berlin Forum to see what I mean.

What can we do to improve the situation?

We can wait that either one of the big players or a clever startup has a great idea. But one of the attractive features of Web 2.0 is user participation. We need more discussions between scientists and software developers on what is needed and what can be done. These discussions are of course already taking place, but science bloggers can do more to collect interesting ideas and articulate them. We want the integration of reference managers in online writing tools such as Google Docs (https://web.archive.org/web/20120611103012/http://docs.google.com/?) or Buzzword (https://web.archive.org/web/20120611103012/http://www.buzzword.com/?), but how do we make our voice heard?

Secondly, we can write applications ourselves. The barriers of entry have become really low, and one reason are the APIs (application programming interfaces) of both science applications or conventional Web 2.0 apps:

  • Scopus (https://web.archive.org/web/20120611103012/http://www.scopus.com/scsearchapi/?)
  • Connotea (https://web.archive.org/web/20120611103012/http://network.nature.com/blogs/user/ianmulvany/2007/08/14/java-wrapper-for-the-connotea-api-now-available?)
  • Google Maps (https://web.archive.org/web/20120611103012/http://code.google.com/apis/maps/?)
  • Google Docs (https://web.archive.org/web/20120611103012/http://code.google.com/apis/documents/overview.html?)
  • YouTube (https://web.archive.org/web/20120611103012/http://code.google.com/apis/youtube/overview.html?)
  • Flickr (https://web.archive.org/web/20120611103012/http://www.flickr.com/services/api/?)
  • Technorati (https://web.archive.org/web/20120611103012/http://technorati.com/developers/api/?)
  • Facebook (https://web.archive.org/web/20120611103012/http://developers.facebook.com/?)
  • eBay (https://web.archive.org/web/20120611103012/http://developer.ebay.com/common/api/?)
  • Amazon (https://web.archive.org/web/20120611103012/http://www.amazon.com/gp/browse.html?node=3435361?)
  • Yahoo (https://web.archive.org/web/20120611103012/http://developer.yahoo.com/?)

And there were hints (https://web.archive.org/web/20120611103012/http://www.ghastlyfop.com/blog/2007/10/facebook-code-on-bebo.html?) of a Nature Network API. With some skills in PHP, Python, Java or Ruby, anybody could create an interesting mashup with these APIs over a weekend. Maybe linking Connotea tags to YouTube videos and Flickr pictures?
